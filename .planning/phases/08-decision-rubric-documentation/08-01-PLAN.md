Create `docs/` directory and `docs/decision-rubric.md`. The document MUST describe the exact logic from code — do NOT invent or simplify. Structure:

**1. Overview** — The v2 ground truth is determined by a weighted multi-signal scoring function, not by category alone. Explain that five context signals are scored and summed into a `weighted_sum`. Thresholds determine the label.

**2. Decision Thresholds** — Document exactly:
- `weighted_sum > +0.30` => THREAT
- `weighted_sum < -0.30` => BENIGN
- `-0.30 <= weighted_sum <= +0.30` => SUSPICIOUS
- `|weighted_sum| < 0.10` => additionally flagged as "ambiguous by design"

**3. Signal Definitions and Scores** — One subsection per signal with exact values:

Zone Type (base scores, modified by sensitivity factor):
- restricted: +0.40, utility: +0.25, perimeter: +0.10, interior: 0.00, parking: -0.15
- Sensitivity factor: `0.6 + (zone_sensitivity - 1) * 0.2` (sensitivity is 1-5 scale)
- Final zone score = base * sensitivity_factor

Time of Day:
- night: +0.35, dawn: +0.15, dusk: +0.10, day: -0.20

Severity:
- CRITICAL: +0.25, HIGH: +0.15, MEDIUM: 0.00, LOW: -0.20
- NOTE: Maximum severity score (+0.25) is below the threshold (0.30), so severity alone NEVER determines GT

Device FPR (linear formula):
- `score = 0.15 - (fpr * 0.4444)` (equivalently: `0.15 - fpr * (0.40 / 0.90)`)
- Low FPR (reliable device) => positive score (threat signal)
- High FPR (unreliable device) => negative score (benign signal)
- Range: FPR=0.05 => +0.13, FPR=0.95 => -0.27

Badge Access:
- Badge scan within 10 minutes: -0.45 (strong benign signal)
- Badge scan within 30 minutes: -0.25 (moderate benign signal)
- No badge data: 0.00 (no contribution)

**4. Scoring Formula** — `weighted_sum = zone_score + time_score + fpr_score + severity_score + badge_score`, rounded to 4 decimal places.

**5. Worked Examples** — Three minimum, drawn from the verified test configurations:

Example 1 — THREAT (T1_canonical_threat):
- Inputs: restricted zone, sensitivity=5, night, FPR=0.15, HIGH severity, no badge
- Zone: +0.40 * (0.6 + 4*0.2) = +0.40 * 1.4 = +0.56
- Time: +0.35
- FPR: 0.15 - (0.15 * 0.4444) = +0.083
- Severity: +0.15
- Badge: 0.00
- Sum: +1.143 => THREAT (well above +0.30)

Example 2 — SUSPICIOUS (S1_ambiguous_by_design):
- Inputs: interior zone, sensitivity=3, dusk, FPR=0.50, MEDIUM severity, no badge
- Zone: 0.00 * (0.6 + 2*0.2) = 0.00
- Time: +0.10
- FPR: 0.15 - (0.50 * 0.4444) = -0.072
- Severity: 0.00
- Badge: 0.00
- Sum: +0.028 => SUSPICIOUS (between -0.30 and +0.30), also ambiguous (|0.028| < 0.10)

Example 3 — BENIGN (B1_canonical_benign):
- Inputs: parking zone, sensitivity=2, day, FPR=0.85, LOW severity, badge 5 min ago
- Zone: -0.15 * (0.6 + 1*0.2) = -0.15 * 0.8 = -0.12
- Time: -0.20
- FPR: 0.15 - (0.85 * 0.4444) = -0.228
- Severity: -0.20
- Badge: -0.45
- Sum: -1.198 => BENIGN (well below -0.30)

**6. Adversarial Cases** — Explain the two key counterintuitive scenarios:

Why HIGH severity can yield BENIGN (B2_adversarial_critical from tests):
- parking zone (sensitivity=1) + day + FPR=0.90 + CRITICAL severity + badge 3 min ago
- Badge access (-0.45) and unreliable device (-0.25) overwhelm severity (+0.25)
- Interpretation: "A known employee badged in 3 minutes ago at a low-sensitivity parking camera that cries wolf 90% of the time — CRITICAL severity label is not credible"

Why LOW severity can yield THREAT (T2_adversarial_low_sev from tests):
- restricted zone (sensitivity=5) + night + FPR=0.10 + LOW severity
- Zone (+0.56) and time (+0.35) overwhelm severity (-0.20)
- Interpretation: "Someone is in a restricted zone at night recorded by a reliable camera — LOW severity from analytics is not reassuring"

**7. Adversarial Signal Injection** — Brief description of `_inject_adversarial_signals`: ~20% of scenarios have one context signal deliberately flipped to create conflicting evidence (SCEN-04 compliance). The flip targets severity, zone, or time+FPR.

**8. Description Pools** — Note that descriptions are drawn from shared pools (ambiguous ~63%, unambiguous-threat ~23%, unambiguous-benign ~14%) and the same description text can appear across all three GT classes in different contexts. Description text is NOT an input to the GT scoring function.
