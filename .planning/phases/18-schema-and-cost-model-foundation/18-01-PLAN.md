    **Structure the document as follows:**

    ## 1. Overview
    Explain that dispatch is a parallel decision alongside GT — it answers "what action should an operator take?" given the classification AND site context. GT alone does not determine dispatch (THREAT at a campus ≠ THREAT at a substation). Dispatch has 5 classes: armed_response, patrol, operator_review, auto_suppress, request_data.

    ## 2. Primary Decision Table (GT × site_type × zone_sensitivity)
    Publish the full decision table. The logic is:

    **For GT = THREAT:**
    - site_type in {substation, solar} OR zone_sensitivity >= 4 → `armed_response`
    - site_type in {industrial, campus} AND zone_sensitivity in {2, 3} → `patrol`
    - site_type == commercial AND zone_sensitivity <= 3 → `patrol`
    - zone_sensitivity == 1 (any site) → `patrol` (low-confidence zone; don't escalate beyond patrol)

    **For GT = SUSPICIOUS:**
    - zone_sensitivity >= 4 → `patrol` (high-sensitivity zone warrants a physical check)
    - zone_sensitivity <= 3 → `operator_review`

    **For GT = BENIGN:**
    - device.false_positive_rate >= 0.70 → `auto_suppress` (unreliable device; suppress)
    - device.false_positive_rate < 0.70 AND recent_zone_events_1h count >= 3 → `request_data` (unusual frequency; gather more)
    - otherwise → `auto_suppress`

    ## 3. Site Threat Multipliers (for cost model)
    Publish the SITE_THREAT_MULTIPLIERS used to scale cost entries:
    | Site type | Multiplier |
    |-----------|-----------|
    | substation | 5.0 |
    | solar | 3.0 |
    | industrial | 2.5 |
    | campus | 2.0 |
    | commercial | 1.5 |

    Explain that these multipliers apply to the THREAT row of the cost matrix (missed THREAT costs more at high-criticality sites). The cost matrix base values are multiplied by the site multiplier for threat-related cells.

    ## 4. Worked Examples
    Provide 5 worked examples — one for each dispatch action — showing the inputs and the resulting dispatch. These must be hand-verifiable so test fixtures can be written against them.

    Example inputs to include:
    - THREAT + substation + sensitivity 5 → armed_response
    - THREAT + commercial + sensitivity 2 → patrol
    - SUSPICIOUS + sensitivity 4 → patrol
    - SUSPICIOUS + sensitivity 2 → operator_review
    - BENIGN + FPR 0.85 → auto_suppress
    - BENIGN + FPR 0.40 + 5 recent zone events → request_data

    ## 5. Cost Matrix (Provisional Benchmark Assumptions)
    Publish the provisional cost matrix that the cost model uses by default. Label clearly as "provisional benchmark assumptions — not sourced from industry data." Include instructions for overriding via --cost-profile JSON.
